D32labels_list / folders_list return null on a freshly-backfilled mirror fixed in PR 5.5/A low

Reported: 2026-05-22 from live Claude Code MCP test · Fixed: PR 5.5/A · Surface: cmd/protonmcp/backfill.go

Root cause: The labels table was populated exclusively through the sync event loop's label.create / label.update handlers. The initial backfill drained messages but never seeded labels, so users whose label set predated their first sync saw {"labels":null} / {"folders":null} until a label-change event happened to fire.

Fix: runBackfill now calls backfillLabels after capturing the event cursor. That calls Session.Client.GetLabels(ctx, LabelTypeLabel, LabelTypeFolder) and upserts each row through store.UpsertLabel. Non-fatal: if the labels fetch fails, the messages backfill still proceeds and a stderr warning surfaces.

D4–D31 added 2026-05-22 from a comprehensive security re-audit covering Phases 1.5/2/3/4 (companion to the original SECURITY.md bootstrap audit). Listed in severity order. Each entry preserves the auditor's evidence / impact / remediation verbatim so the rationale survives a close.

D4Touch ID helper substitutable via PROTONMCP_TOUCHID env var open critical

Reported: 2026-05-22 (security re-audit) · Surface: internal/approval/path.go:24, cmd/protonmcp/serve_stdio.go:138-154

Evidence: resolveHelperPath honors PROTONMCP_TOUCHID as its first candidate and was never gated to test-only. serve_stdio calls approval.ResolveHelperPath(os.Args[0]) without stripping the env first. Whatever process spawns protonmcp serve-stdio (Claude Desktop today, anything else tomorrow) controls the env it hands us.

Impact: A malicious or compromised parent process can set PROTONMCP_TOUCHID=/bin/true (or any exit-0 binary) before launching us; the broker then "approves" every prompted tool — including mail_send, mail_forward, mail_delete_permanent — with no Touch ID interaction at all. The middleware happily records approval_source=touchid for what is in fact zero authentication. Same vector applies to mail_delete, folders_delete, anything prompt+confirm:true.

Remediation: Only honor PROTONMCP_TOUCHID when testing.Testing() is true (Go 1.21+), or behind a build tag, or refuse the env var entirely in serve-stdio mode. At minimum, log loudly when the env override is in effect and exclude it from secretEnvNames-style cleanup.

Related: new (post-Phase-4 broker)

D5PROTONMCP_DEBUG=1 in serve-stdio dumps refresh tokens to stderr open critical

Reported: 2026-05-22 (security re-audit) · Surface: internal/proton/client.go:331-340, cmd/protonmcp/serve_stdio.go (no guard)

Evidence: NewManager enables the debug transport unconditionally on os.Getenv("PROTONMCP_DEBUG") != "". The original SECURITY re-audit explicitly called this out as "pre-Phase-3 invariant" (updated foundational #6): "refuse to start in daemon mode with PROTONMCP_DEBUG=1." That recommendation was never implemented. redactDump reduces but does not eliminate leak risk — its body-token regex is a fixed list of JSON field names, and the dumpHeaderRedactRE is similarly enumerated; new endpoints / new field names slip through.

Impact: Claude Desktop captures spawned-MCP-server stderr into its own log directory. Any user who sets PROTONMCP_DEBUG=1 (or has it inherited from their shell) and runs the MCP integration will spray Proton API traffic — including AuthInfo/SRP exchanges, full message decryption-precursor metadata, and any future-added header/field the redactor doesn't yet know about — into a path the user does not realize is sensitive.

Remediation: In runServeStdio, os.Unsetenv("PROTONMCP_DEBUG") before constructing the manager, or hard-error if it's set. Document that PROTONMCP_DEBUG is for whoami/backfill-style interactive shells only.

Related: SECURITY.md updated-foundational #6 (still open)

D6allowed_recipients enforcement bypassed for mail_reply, mail_reply_all, mail_send_draft fixed in PR 5.5/A high

Reported: 2026-05-22 (security re-audit) · Surface: internal/mcptools/mail_send.go:104, 148, 190, internal/mcp/middleware.go:179-186

Evidence: Three send-family tools explicitly set Recipients: nil. The middleware (middleware.go:179) only runs the allowlist check when t.Recipients != nil. Comment in the code acknowledges this and defers validation to "the handler" — but the handler proceeds to fetch the parent, build the recipient list, and call SendDraft without checking the policy's AllowedRecipients. So an allowed_recipients: ["@mycompany.com"] policy on mail_send is enforced, but the LLM can reply to any message whose original sender is outside that domain and exfiltrate to that sender — the allowlist is silently void.

Impact: An attacker-crafted email arriving in the user's inbox, combined with a prompt-injection success (mail_read is allow-by-default), can cause mail_reply with attacker-controlled body to attacker's address. The user's policy override stating "only send to @mycompany.com" is bypassed, with a Touch ID approval that says only "Reply to message X (recipient = original sender)" — no recipient address shown.

Remediation: In the reply/forward/send_draft handlers, after resolving recipients but before SendDraft, run the same firstDisallowedRecipient check against the policy's AllowedRecipients. Either re-resolve the policy inline or pass it through Context.

Related: new (Phase 5/D)

D7Recipient allowlist uses raw-string suffix matching — bypassable via display names / CSV smuggling fixed in PR 5.5/A high

Reported: 2026-05-22 (security re-audit) · Surface: internal/mcp/middleware.go:26-56, internal/mcptools/mail_send.go:277-287

Evidence: firstDisallowedRecipient does strings.HasSuffix(lower, "@example.com") over the raw ["alice@example.com"] strings, no email parsing. extractSendRecipients is json.Unmarshal + slice append, no normalization. Inputs like "Alice <evil@attacker.com>" end with > and fail the @example.com suffix check (false negative — denies legitimate display-name forms). More concerning, a recipient list ["evil@attacker.com,alice@allowed.com"] (CSV in one slot) gets HasSuffix(..., "@allowed.com") = true and passes the gate.

Impact: False negatives (legit display names blocked) erode the feature's usability; false positives (CSV smuggling) erode the feature's security guarantee. The allowlist is the last line of defense before a Touch ID approval that — for the basic send case — at least lists the recipients literally, so a careful user might catch it; the broker has no second-line defense.

Remediation: In firstDisallowedRecipient, parse every entry via mail.ParseAddress and run the suffix check against the resulting .Address. Return any unparseable entry as disallowed. Same change in extractSendRecipients.

Related: new (Phase 5/D)

D8Audit Begin/Complete run under request ctx — cancellation drops outcome fixed in PR 5.5/C high

Reported: 2026-05-22 (security re-audit) · Surface: internal/audit/audit.go:107-184, internal/mcp/middleware.go:140-143

Evidence: Begin uses db.ExecContext(ctx, ...), Complete and SetDecision likewise. The middleware's defer (middleware.go:140-142) calls m.audit.Complete(ctx, ...) with the same context that may have been cancelled mid-handler (handler timeout, client disconnect, SIGTERM). When ctx is already Done, modernc/sqlite's ExecContext returns context.Canceled immediately, the audit row stays at NULL outcome forever, the JSONL line never appends.

Impact: The audit log loses provenance precisely when it matters most — during forced shutdowns / Ctrl-C of long-running write tools. A mail_send that the user aborts mid-Touch-ID via SIGTERM leaves an audit row with NULL outcome (so the forensic question "did this actually send?" can't be answered from the audit). The audit_log is supposed to be the source of truth; this design makes it forgetful exactly when the daemon's behavior gets interesting.

Remediation: In Complete (at minimum) and Begin (defensible), use a detached short-timeout context — same pattern as detachedShutdownCtx() in internal/proton/client.go:105. The audit write should always run to completion (or its own short timeout), not inherit the request context.

Related: M-4/M-5 / Foundational #5 spirit (new specific instance)

D9Policy reload PID file unauthenticated — any local process can SIGHUP via planted file open high

Reported: 2026-05-22 (security re-audit) · Surface: internal/policy/pidfile.go:77-93, cmd/protonmcp/policy.go:36-60

Evidence: ReadPIDFile reads ~/Library/Application Support/protonmcp/serve-stdio.pid, parses an integer, calls processAlive(pid) (which is kill -0), and policy.go:55 then p.Signal(syscall.SIGHUP) to that PID. The only check is "is the PID alive?" — no equivalent of "is this PID's binary actually protonmcp?", no UID match check on the target process. A malicious local process (or a corrupted file) can write any PID into that path and the protonmcp policy reload CLI will SIGHUP it.

Impact: Weaponized to SIGHUP unrelated processes the user owns — many of which treat SIGHUP as "terminate gracefully" (sshd children, screen sessions) or as "reload config" (which can be dangerous if the config has been concurrently corrupted). Local-process attack, no privilege escalation, but a real footgun.

Remediation: Before signaling, read /proc-equivalent (libproc proc_pidpath is already wrapped in internal/caller/caller_darwin.go) for the target PID; verify the path matches os.Executable() of the protonmcp CLI. Also verify the file is owned by os.Geteuid(). The advisory flock on the file at write time doesn't protect a read-side reader at all.

Related: new (Phase 4/A) · Needs verification: kill(2) requires sender UID == target UID for non-zero signals — bounded to same-user processes, not a privilege escalation.

D10tryResume and runLogout never stored.Zero() — salted key material leaks to GC fixed in PR 5.5/D high

Reported: 2026-05-22 (security re-audit) · Surface: cmd/protonmcp/session.go:109-147, cmd/protonmcp/login.go:64-79

Evidence: Re-audit #2's B-3 partial. inspect.go:44 correctly defer stored.Zero()s. The hot paths — tryResume (called by every whoami/backfill/sync/read/search) and runLogout — still don't. Pattern stored, err := keystore.Load() followed by no zero. SaltedKeyPass is ultimately copied into the Session via Resume, but the local Live struct's SaltedKeyPass Secret holds its own slice header that points to bytes the keystore decoded — those bytes stay live until GC.

Impact: Heap-resident PGP-key-deriving salted material for the full lifetime of the process for every short-lived CLI run. A coredump / memory-image attack recovers it. The mitigation exists in inspect.go and is exactly two lines per call site.

Remediation: Add defer stored.Zero() at cmd/protonmcp/session.go:113 and cmd/protonmcp/login.go:69. Verify by grep -n "keystore.Load" cmd/.

Related: B-3 (still open after Phase 1.5)

D11raw_json truncation produces invalid JSON open high

Reported: 2026-05-22 (security re-audit) · Surface: internal/proton/messages.go:108-110

Evidence: Re-audit #2 flagged this verbatim: if len(raw) > MaxRawJSONBytes { raw = append(raw[:MaxRawJSONBytes], []byte("...\"[truncated]\"")...) }. The result is ...{actual JSON prefix}...\"[truncated]\" — broken JSON. mail_list_attachments (internal/mcptools/attachments.go) doesn't currently parse raw_json, but any future consumer that does json.Unmarshal([]byte(row.RawJSON), ...) panics or errors on the corrupted bytes. There's also no slog.Warn on truncation, so silent corruption.

Impact: A malicious or buggy server returning >1 MiB metadata blobs poisons rows in a way that breaks downstream consumers later. The intent of the cap was to prevent DoS; the implementation also breaks correctness for that mailbox forever (the row stays poisoned until next backfill).

Remediation: Replace with raw = []byte({"truncated": true, "original_bytes": <n>, "id": "<m.ID>"}). Log a Warn with msg_id and bytes. Better: enforce a hard cap on per-row write to SQLite at the store layer, return an error to the upserter.

D12DSN pragma injection via --db flag open medium

Reported: 2026-05-22 (security re-audit) · Surface: internal/store/store.go:115-129

Evidence: Re-audit #2 confirmed B-6 NOT FIXED and the code is unchanged. buildDSN returns path + "?" + v.Encode() — path is unescaped. --db '/tmp/x.db?_pragma=key("...")&_pragma=cipher_compatibility(3)' injects extra pragmas; the resulting DSN looks like /tmp/x.db?_pragma=key("...")&_pragma=...?_pragma=busy_timeout(5000)&... which modernc/sqlite parses to honor the attacker's pragmas alongside ours.

Impact: Today's pragmas of concern are limited (no SQLCipher integration → key(...) is a no-op), but journal_mode(off) would silently disable WAL/durability, and secure_delete(off) would defeat C-1's mitigation. Attack surface = anyone who can supply --db (any subcommand, including via shell completion / aliases).

Remediation: Build the DSN via url.URL{Scheme: "file", Opaque: path, RawQuery: v.Encode()}.String() or url.PathEscape(path). Reject paths containing ? outright. Test added in store_test.go.

Related: B-6 (still open after two cycles)

D13Plaintext bodies persisted indefinitely — no eviction, no purge command fixed in PR 5.5/B medium

Fix (PR 5.5/B): Two layers. (1) New protonmcp purge [--older-than D] [--dry-run] [--vacuum] subcommand hard-deletes body_text / body_html / body_cached_at on rows whose body cached-at is past the cutoff. The messages_fts_update trigger fires on the same UPDATE so the FTS index drops the purged content too; combined with the secure_delete=on pragma already set in store.go, freed cells are zeroed on the next page write rather than left as recoverable slack space. --vacuum forces it immediately. (2) serve-stdio sweeps at startup with the default 30-day retention (store.DefaultBodyRetention); failures log Warn but don't block the daemon. Closes the audit cycle's biggest residual. Pending Phase-6 envelope encryption.

Reported: 2026-05-22 (security re-audit) · Surface: internal/store/messages.go:286-357, internal/proton/body.go:38-88

Evidence: Re-audit #2's C-1 — the single largest residual risk — is unchanged. BodyTTL = 24h (line 286) gates only GetCachedBody reads; stale rows return ErrNotFound but the body_text/body_html columns are not deleted. InvalidateBodyCache (line 272) only nulls body_cached_at, leaving the plaintext columns intact. No startup sweeper, no scheduled deletion, no protonmcp purge. Phase 5 added draft bodies to the same persistence path, so the surface grew.

Impact: Every message the user has ever opened is recoverable in cleartext from store.db indefinitely, without the Keychain. Time Machine, iCloud Documents sync, Spotlight indexing, support-image disposal all bypass the 0600 perms. Drafts (containing the user's outgoing prose) are now included. secure_delete=on (B-10) is in place but never triggered because no DELETE ever runs.

Remediation: Add a startup sweep UPDATE messages SET body_text = NULL, body_html = NULL WHERE body_cached_at IS NULL OR body_cached_at < ?. Ship protonmcp purge [--all|--older-than DUR]. Document loudly in README. Long-term: SQLCipher or envelope-encrypt the body columns with a Keychain-stored key.

D14Approval cache key omits Confirm flag — policy reload doesn't invalidate stale approvals fixed in PR 5.5/C medium

Reported: 2026-05-22 (security re-audit) · Surface: internal/approval/cache.go:58-66, internal/approval/broker.go:94-110

Evidence: cacheKey hashes tool || pid || args — not the confirm:true bit, not the TTL, not the allowed_recipients. If a user has approved mail_move (no confirm) with TTL=5m, then SIGHUPs an updated policy that flips it to confirm:true, the still-valid cache entry from the original approval will satisfy a subsequent call without running the NSAlert step. Same shape: changing allowed_recipients doesn't invalidate cached approvals for the new list.

Impact: Policy reload doesn't fully take effect for in-flight cached approvals — a 5-minute window where the prior, possibly weaker, decision keeps applying. The model says reload is supposed to be "atomic" (policy_test.go's TestReloadAtomicOnInvalidYAML); approval cache breaks that for the approval pathway.

Remediation: Include confirm and a hash of AllowedRecipients in the cache key. Or, simpler: drop the entire approval cache on engine.Reload() — call from serve_stdio.go's HUP handler.

D15Approval subCtx.Err() uses == — misses wrapped deadline errors fixed in PR 5.5/D medium

Reported: 2026-05-22 (security re-audit) · Surface: internal/approval/broker.go:170-176

Evidence: if subCtx.Err() == context.DeadlineExceeded { ... return ErrUserCanceled }. Go's context.WithTimeout returns the deadline error directly (so == works today), but any wrapping (via a future custom Context, or a child layer) breaks this. D19looksLikeToken heuristic misses URL-encoded / quoted-context tokens fixed in PR 5.5/D medium

Reported: 2026-05-22 (security re-audit) · Surface: internal/redact/redact.go:154-171

Evidence: The heuristic immediately returns false for any string containing space, /, :, ", ', <, >, \n, \t. That's intentional ("phrase-shaped, not token-shaped") but means "refresh_token=eyJhbGciOi…" (embedded =, OK), "Bearer eyJhbG…" (space → not redacted), "<token>eyJhbG…</token>" (angle brackets → not redacted), "prefix: eyJhbG…" (colon → not redacted) all pass through. A library error like unknown auth response: {"refreshToken":"eyJhbG…"} passes through completely.

Impact: Error messages from go-proton-api that quote response bodies into their Error() string can leak tokens to slog despite the value heuristic. Combined with slog.Warn("…", "err", err.Error()) pattern (multiple call sites: session.go:78,99,142,181, login.go:66, etc.), this is a real leak vector the way M-4 was originally framed.

Remediation: In addition to the existing token shape, scan for JWT regex \b[A-Za-z0-9_-]{8,}\.[A-Za-z0-9_-]{8,}\.[A-Za-z0-9_-]{8,}\b and Proton's known token shape (long base64url runs ≥40 chars) across the whole value and selectively replace those substrings. M-4 is also still architecturally open: %w wrapping of gopenpgp/library errors is the upstream cause.

D20caller.Resolver records daemon's own UID, not spawner's — audit field misleading open medium

Reported: 2026-05-22 (security re-audit) · Surface: internal/caller/caller.go:55-63

Evidence: Resolve() sets Caller.UID = os.Getuid() — that's the daemon's own UID, not the parent's. Comment at the top of the file claims this is acceptable because "our parent PID can't legitimately change after startup" — correct for PID but the UID field documents the wrong fact entirely.

Impact: Audit log's caller_uid column is uninformative (always equal to the daemon's UID). In Phase 6 (Unix socket multi-client), this becomes actively wrong — the peer over SO_PEERCRED may have a different UID and we'd record our own anyway. Defense-in-depth: a per-call UID-mismatch check makes no sense with this field. Today's MCP stdio model trusts the spawner; the audit field should still record the spawner's UID, not ours.

Remediation: On macOS use libproc.proc_pidinfo(PROC_PIDTBSDINFO) (or kvm/sysctl) to get the parent process's real UID. On Linux read /proc/<ppid>/status. Until that's wired, document the field as "daemon UID" and rename for clarity.

D21Touch ID prompt body un-sanitized — NSAlert spoofing via unicode/length fixed in PR 5.5/A medium

Reported: 2026-05-22 (security re-audit) · Surface: internal/approval/broker.go:119-138, internal/mcptools/mail_send.go:294-307

Evidence: sendPromptBody builds body from in.To, in.CC, in.BCC, in.Subject joined with newlines, where every field is the LLM-supplied JSON. JSON is passed verbatim through json.Marshal to the Swift helper, which sets it as alert.informativeText. NSAlert accepts arbitrary length and renders newlines literally — a malicious-LLM-supplied subject doesn't render terminal escapes, BUT a multi-megabyte body can hang the helper / trigger AppKit OOM, and unicode tricks (RTL override ‮, zero-width joiners, Apple system-font lookalikes) can mislead the user about what they're approving.

Impact: NSAlert spoofing — the user sees a prompt that says "To: [a bunch of unicode that visually reads 'alice@trusted.com' but is actually 'mallory@evil.com']". Touch ID approval still works on the underlying data, the user sees a falsified-looking dialog.

Remediation: In sendPromptBody, cap each field length (e.g. 200 chars), strip control chars (sanitize.stripControlChars is right there), and apply unicode normalization (norm.NFKC + bidi-control stripping). Same hardening for Title.

D24No integrity check on the running protonmcp binary — PATH swap attack open medium

Reported: 2026-05-22 (security re-audit) · Surface: cmd/protonmcp/serve_stdio.go:32-34

Evidence: serve-stdio's trust model: "the trust boundary is whoever spawned this process which inherits our UID." Claude Desktop spawns us via claude_desktop_config.json → {"command": "/abs/path/to/protonmcp", "args": ["serve-stdio"]}. But runInstall writes binPath from os.Executable(), which on macOS may resolve through a symlink chain — and if the user's installation later moves the binary, the recorded path may become invalid or, worse, point at a newly-planted binary. There's no integrity check (no path signature, no SHA recorded). Until Phase 7 code signing this is the entirety of the trust chain.

Impact: A swap-attack scenario where an attacker who can write to the recorded command path can run as the MCP server with full Keychain access. Defense-in-depth gap that the docs treat as out of scope.

Remediation: Short-term, document loudly that claude_desktop_config.json paths must point at signed binaries (chicken-and-egg with Phase 7). Medium-term, on every serve-stdio startup, log the SHA-256 of the running binary and require a ~/Library/Application Support/protonmcp/expected_sha file match to proceed.

Related: new (Phase 6/7-adjacent)

D1mail_list — folder="all" sorts oldest-first fixed

Reported: 2026-05-22 · Fixed: PR #37 · Surface: mail_list

Root cause: primaryFolder() in internal/proton/messages.go returned the literal string "all" as a fallback for messages without any system folder label. So messages.folder = 'all' was a real bucket — the user's older messages without system labels. When the LLM passed folder="all" expecting "all folders", the SQL filtered to that bucket and returned a small set of older messages.

Fix: primaryFolder now returns "" for the no-system-label case. mail_list folder="all" is recognised as a synonym for "no folder filter" (along with any / all_mail / *). The DSL also treats in:all the same way. Existing rows with the literal "all" value are migrated to "" by migration 0002_folder_all_to_empty.sql.

D3mail_search — since/until silently ignored fixed

Reported: 2026-05-22 · Fixed: PR #37 · Surface: mail_search

Root cause: Two layers. The MCP input schema for mail_search only had query / limit / cursor — top-level since / until were silently stripped. When the LLM embedded date filters in the query string instead, our DSL only recognised after: / before:, so since: / until: tokens fell through to the unknown-prefix path (internal/store/query.go:80) and became bare FTS terms that matched nothing.

Fix: Added top-level since / until parameters to mail_search's input schema (mirrors mail_list's shape) and aliased since: → after: + until: → before: in the DSL parser so embedded date filters work too.
