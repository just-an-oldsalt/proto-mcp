D1mail_list — folder="all" sorts oldest-first fixed

Reported: 2026-05-22 · Fixed: PR #37 · Surface: mail_list

Root cause: primaryFolder() in internal/proton/messages.go returned the literal string "all" as a fallback for messages without any system folder label. So messages.folder = 'all' was a real bucket — the user's older messages without system labels. When the LLM passed folder="all" expecting "all folders", the SQL filtered to that bucket and returned a small set of older messages.

Fix: primaryFolder now returns "" for the no-system-label case. mail_list folder="all" is recognised as a synonym for "no folder filter" (along with any / all_mail / *). The DSL also treats in:all the same way. Existing rows with the literal "all" value are migrated to "" by migration 0002_folder_all_to_empty.sql.

D2mail_list — since filter returns empty on folder="all" fixed

Reported: 2026-05-22 · Fixed: PR #37 · Surface: mail_list

Root cause: Same as D1 — once folder="all" filtered to the small "no system label" bucket, adding since="2026-05-01" intersected with a bucket whose messages were all older than May 2026, yielding empty results.

Fix: Closed by D1's fix. folder="all" no longer narrows to the synthetic bucket, so since applies across the whole mailbox as expected.

D3mail_search — since/until silently ignored fixed

Reported: 2026-05-22 · Fixed: PR #37 · Surface: mail_search

Root cause: Two layers. The MCP input schema for mail_search only had query / limit / cursor — top-level since / until were silently stripped. When the LLM embedded date filters in the query string instead, our DSL only recognised after: / before:, so since: / until: tokens fell through to the unknown-prefix path (internal/store/query.go:80) and became bare FTS terms that matched nothing.

Fix: Added top-level since / until parameters to mail_search's input schema (mirrors mail_list's shape) and aliased since: → after: + until: → before: in the DSL parser so embedded date filters work too.
