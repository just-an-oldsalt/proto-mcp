Quickstart for whoever picks this up

# build
go build -o bin/protonmcp ./cmd/protonmcp

# fully interactive — prompts for whatever it needs, passwords are
# echo-off via /dev/tty, TOTP and mailbox password are asked only if
# the server actually requires them
./bin/protonmcp whoami

# scripted — env vars override interactive prompts
PROTONMCP_EMAIL=you@proton.me \
PROTONMCP_PASSWORD='...' \
PROTONMCP_TOTP=123456 \
  ./bin/protonmcp whoami

Successful output prints user ID, primary address, address count, password mode, 2FA mode, storage used, and login+unlock duration.

Phase 1 — Local SQLite mirror in progress

Goal: cold backfill of message metadata into a local SQLite store so reads and search can be answered offline.

  • [x] Picked pressly/goose/v3 for migrations (embedded via //go:embed), modernc.org/sqlite for the driver — pure Go, no cgo, FTS5 built in.
  • [x] internal/store package: Open(path) returns a *Store wrapping *sql.DB, applies migrations, sets WAL + foreign_keys pragmas. Default path is ~/Library/Application Support/protonmcp/store.db (override via XDG_DATA_HOME).
  • [x] Schema implemented in 0001_initial.sql: messages, message_labels (FK + cascade), labels, messages_fts (FTS5 with insert/update/delete triggers), sync_state, audit_log. Tests cover insert/get, FTS5 search, cascade delete, label replacement, sync_state round-trip.
  • [x] proton.Session.ForEachMessageMetadataPage walks pages via client.GetMessageMetadataPage (page size 150, the API cap) and yields batches to a callback. ToStoreMessage translates MessageMetadata into the store row shape.
  • [x] Cold backfill command: protonmcp backfill — interactive login, capture event cursor first, count messages, confirm if over 5000 (overridable via --yes), drain into store with progress reporting every 2s. Supports --limit N for spot-checks.
  • [x] Sync cursor: captured via client.GetLatestEventID before drain (so events during backfill aren't lost — sync replays them later), persisted under sync_state.event_cursor on completion.
  • [ ] Smoke-test against rdort@proton.me — run ./bin/protonmcp backfill --limit 100 first, confirm rows land, then full drain.
  • [ ] Skip body fetch in this phase. Body decryption is in Phase 2. Just metadata: envelope fields, label IDs, folder, flags.
  • Known limitation: Bridge's SDK doesn't expose Proton's ConversationID, so thread_id is set to the message ID for now. Phase 2 will reconstruct threads from RFC 2822 In-Reply-To / References headers once we start fetching full messages.

Gotchas

  • The page size for GetMessageMetadataPage is API-capped at 150; loop with the right cursor or filter.
  • Some accounts have hundreds of thousands of messages. Print a count first and confirm before draining — backfill is the only operation that scans the whole mailbox.

Phase 2 — Sync loop + lazy decrypt + FTS done

Landed: #19 sanitize, #20 body-fetch + cache + read CLI + audit folds (B-1 / B-3 / B-4 / B-10 / B-2 / B-12), #21 search DSL + FTS + search CLI, #22 event-loop sync + sync CLI + B-9 backfill bounds. Pre-Phase-2 hotfix #18 covered B-5 / B-7 / B-8 / B-11.

Phase 1 left us with a populated messages table (envelopes only) and a stored event_cursor. Phase 2 closes the loop: keep the local mirror in sync with the server via event-id polling, decrypt + cache message bodies on demand, and turn the FTS5 schema we already built into an actual search API. Split into focused branches, same pattern as Phase 1.5.

Branches in this phase

2/A — phase-2/html-sanitize. Foundation; no other deps.

One policy tuned for LLM consumption: preserve readable structure (<p>, <br>, <b>, <i>, <a> with stripped href, headings, lists, blockquotes), strip everything else. Always remove <script> / <iframe> / remote image src regardless of policy permissions.

  • [ ] Table-driven tests for each strip rule + a fixture set of real-world Proton email HTML.

2/B — phase-2/body-fetch. Depends on 2/A.

  • [ ] proton.Session.FetchAndDecryptMessage(ctx, msgID): calls client.GetFullMessage (verify exact name in the SDK), decrypts the body using Session.AddrKRs[message.AddressID], returns {text, html, headers, attachments_meta}. HTML goes through internal/sanitize before return.
  • [ ] store.UpdateBodyCache(ctx, msgID, text, html) plus store.GetCachedBody(ctx, msgID) with a TTL check — 24h per the design spec, as defense-in-depth on top of the event loop's invalidation.

2/C — phase-2/sync-loop. Independent of A/B.

  • [ ] internal/sync package with RunOnce(ctx, sess, store): reads event_cursor from sync_state, calls client.GetEvent in a loop until the server says it's caught up, applies each diff to the store (CRUD on messages, label-set updates, folder moves), advances the cursor as it goes (so a crash mid-loop doesn't re-process events).
  • [ ] Handle the REFRESH instruction (the server returns this when our cursor is too old) by falling through to a full backfill + fresh cursor capture.
  • [ ] Bounded retries with exponential backoff on network errors. Stop and surface after N consecutive failures — no infinite retry loops that could trip Proton anti-abuse.
  • [ ] protonmcp sync CLI subcommand that invokes RunOnce. Useful for testing now; Phase 6's daemon will call this on a 30s/5m active/idle cadence from a goroutine.
  • [ ] Token rotation via the existing AuthHandler chain — sync is just one more API caller, so the cookies + rotated tokens persistence work in 1.5/C already covers it.

2/D — phase-2/search. Independent of A/B/C.

  • [ ] internal/store.Search(ctx, query, limit, cursor) on top of the existing messages_fts table (no schema change needed — Phase 1 already wired the triggers).
  • [ ] Small query-DSL parser supporting from:, to:, subject:, in:<folder>, before:<date>, after:<date>, has:attachment, plus bare terms (full-text MATCH against subject + body). Prefixes lower to structured-column predicates; bare terms go into the messages_fts MATCH expression.

Order & dependencies

  • Land 2/A first — it's the dependency for 2/B and the eventual Phase 3 MCP layer.
  • 2/B, 2/C, 2/D can land in any order after A. Doing 2/B next gives the most user-visible win (can actually read mail).
  • Recommended sequence: A → B → D → C. B + D both cement single-process flows; C introduces the long-running "apply diffs" concept that bleeds into Phase 6's daemon work.

Decisions from sign-off

  • Q1: Strict policy. bluemonday with a minimal allowlist (p, br, b, i, a, ul, ol, li, h1-h4, blockquote). Everything else stripped. Scripts / iframes / remote images stripped unconditionally. Prompt-injection risk dominates; LLM reads either format fine.
  • Q2: Simple In-Reply-To chasing. Document the mailing-list limitation; full RFC 5322 / jwz threading is its own follow-up if it bites in practice.
  • Q3: Yes, ship protonmcp sync as a one-shot CLI. Phase 6 daemon will call into the same RunOnce.
  • Q4: Full DSL. from / to / subject / in / before / after / has:attachment + bare FTS5 terms.
  • Q5: JSON output for protonmcp read. Same shape Phase 3's MCP server will return.

Gotchas inherited from the original spec

  • HTML body must be sanitized before returning to the LLM. Use microcosm-cc/bluemonday. Strip <script>, <iframe>, remote image src (tracking pixels). Implemented in 2/A.
  • Snippets: Proton-side snippets can be empty. Generate locally from decrypted body text — first 200 chars, collapse whitespace, strip quoted-reply markers (> ). Implemented in 2/A and reused by 2/D.

Phase 3 — MCP server (read tools only) planning

Hand-rolled JSON-RPC 2.0 over stdio per the MCP 2025-06-18 spec. Same shape as Phase 2 planning: focused sub-branches with explicit dependencies and a small list of decisions for sign-off.

Framing — settled from the spec, not an open question

The MCP spec is definitive on stdio framing, so I'm folding the old open question into the plan:

  • Messages are newline-delimited JSON. Each line is one complete JSON-RPC request, notification, or response. No embedded newlines inside messages.
  • stdout is reserved for MCP traffic — anything else (debug prints, third-party library chatter) corrupts the stream.
  • stderr is the only allowed log surface. Phase 1.5/D's slog writer already targets stderr, so nothing to wire there.

Branches in this phase

3/A — phase-3/mcp-framing. Foundation; no other deps.

  • [ ] internal/mcp package with the JSON-RPC 2.0 base: Request / Response / Error types, NDJSON read/write over io.Reader / io.Writer, robust to framing errors (a malformed line returns a JSON-RPC parse-error response without killing the loop).
  • [ ] Server struct with a map[string]Tool registry. A Tool bundles name / description / inputSchema / optional outputSchema / handler func. Registration is static at server-construction time.
  • [ ] initialize / notifications/initialized handshake. Declares the tools capability with listChanged: false for v1.
  • [ ] tools/list and tools/call dispatch. tools/list returns the registered Tool definitions verbatim; tools/call looks up the handler by params.name and runs it.
  • [ ] Error mapping — two flavors per spec:
    • JSON-RPC error field for protocol failures (unknown tool, malformed args, server-internal).
    • result.isError: true with a text content for tool-execution failures (decrypt failed, message not found, etc.). The LLM sees an error message instead of falling out of the conversation.
  • [ ] Tests with an in-memory pipe pair — a fake stdio loop driven by table fixtures. No live API needed.

3/B — phase-3/read-tools. Depends on 3/A.

  • [ ] account_whoami — wraps the existing whoami output (email, addresses, plan info), JSON body in a text content block.
  • [ ] mail_list — paginated listing from store.messages. Params: folder, label_id, limit, cursor, unread_only, since, until (matches the design spec's tool schema). Returns envelope summaries + next cursor.
  • [ ] mail_search — wraps Phase 2/D's store.Search. Translates the existing query DSL parser; result shape matches mail_list.
  • [ ] mail_read — wraps Phase 2/B's FetchAndDecryptMessage + body-cache check. Both text and html in the output (sanitized via internal/sanitize, which is the boundary enforcement of Foundational #7).
  • [ ] mail_read_thread — selects messages by thread_id, sorted oldest-first. Returns each message body via the same cache-or-fetch path as mail_read.
  • [ ] mail_list_attachments — parses messages.raw_json for attachment metadata (id / filename / mime_type / size / inline). Pure DB read, no network. mail.download_attachment stays in Phase 4/5 since it has a prompt policy.
  • [ ] labels_list and folders_list — split by labels.type column (type=1 → labels, type=3 → folders).
  • [ ] mail_sync — thin wrapper around internal/sync.RunOnce. Tool description steers the LLM to call it for recency-sensitive queries; default behavior for everything else is read-from-mirror. See Q7 decision.
  • [ ] Each tool gets a JSON-schema inputSchema and an outputSchema. Both are hand-written in Go literals next to the handler — refactor to reflection-based generation only if maintenance bites.

3/C — phase-3/serve-stdio. Depends on 3/A + 3/B.

  • [ ] protonmcp serve-stdio subcommand. Acquires a session eagerly at startup (we'd rather fail loudly on broken auth than have the first mail_read surprise the LLM with a 30-second SRP login). Spawns the NDJSON loop.
  • [ ] Foundational #3 (MCP trust model):
    • stdio is the only transport. No net.Listen("tcp", ...) anywhere in the binary.
    • init()-time assertion that panics if any test or future code tries to start a TCP listener — locks the invariant before it can regress silently.
    • No SO_PEERCRED check needed in stdio mode; the trust boundary is "whoever spawned this process," which inherits our UID and is our threat model's accepted trust.
  • [ ] Prompt-injection runtime banner — first-call tools/list response includes a top-level advisory in the description for mail_read / mail_search noting that email content is untrusted input and the LLM should treat instructions inside email bodies as data, not commands. Not a substitute for the Phase 4 policy engine but a useful breadcrumb.
  • [ ] protonmcp install writes a Claude Desktop config snippet (~/Library/Application Support/Claude/claude_desktop_config.json) with the absolute path to the binary. Idempotent — re-running updates the path rather than duplicating the entry.
  • [ ] End-to-end smoke: launch via Claude Desktop, call tools/list, mail_search, mail_read. Capture transcript to docs/.

Order & dependencies

A first (foundation), then B and C can land sequentially. B is the bulk of the work (8 tools × small handlers). C is small but it's the integration moment — easier to land after B is solid.

Decisions locked from sign-off

  • Q1: Tool naming = dots. mail_list, mail_search, mail_read, etc. Matches the design spec's namespace look. JSON-RPC method-name is still tools/call (MCP convention); the dotted name is the params.name value.
  • Q2: Result shape = structured. Every tool returns content: [{type:"text", text:""}] AND a parallel structuredContent object. Every tool declares an explicit outputSchema. Boilerplate accepted in exchange for client validation affordances.
  • Q3: Cursors = opaque. base64(json{offset, query_hash}). Server validates query_hash matches the current query before honoring the offset; mismatch returns a protocol error rather than silently misinterpreting.
  • Q4: Session acquire = eager. SRP / Keychain resume runs during initialize. Auth failures land as "MCP server failed to start" in Claude Desktop, not as mid-conversation tool errors.
  • Q5: mail_read_thread = oldest-first. Conversation order.

FIDO2 / passkey 2FA support

The current code only handles TOTP. If an account has FIDO2 + TOTP, we fall through to TOTP transparently. If an account has FIDO2 only, login errors out with a pointer to add TOTP in Proton settings as a stop-gap.

Implementing native FIDO2 means:

  • Parse the WebAuthn challenge from auth.TwoFA.FIDO2.AuthenticationOptions (it's any in the SDK — likely a JSON-encoded PublicKeyCredentialRequestOptions).
  • Drive the authenticator. On macOS this means either:
    • Platform authenticator (Touch ID / passkey via iCloud Keychain): use the AuthenticationServices framework (ASAuthorizationController with ASAuthorizationPlatformPublicKeyCredentialAssertionRequest). Requires a small Swift helper similar to the planned Touch ID prompt helper.
    • Roaming authenticator (USB / NFC security key like YubiKey): shell out to fido2-assert from libfido2 (Homebrew: brew install libfido2), or link against it via cgo.
  • Submit the assertion as proton.FIDO2Req to client.Auth2FA(ctx, Auth2FAReq{FIDO2: ...}). Field mapping: ClientData, AuthenticatorData, Signature, CredentialID (note: CredentialID in the SDK is []int, probably a byte-by-byte JSON encoding — verify against the SDK before sending).

Estimated effort: ~1 day for platform-authenticator only, ~2-3 days for roaming key support too. Probably worth bundling with the Touch ID approval helper from Phase 4 since both want a Swift helper that drives LocalAuthentication / AuthenticationServices.

Discovered gotchas (keep this list growing)

  • go-proton-api ships with a replace directive pointing resty at github.com/ProtonMail/resty/v2. Replace directives don't propagate from dependencies — must be mirrored in our go.mod. Without it, builds fail on missing resty.MultiPartStream / NewByteMultipartStream.
  • WithAppVersion is validated server-side against a known allowlist. Random strings get rejected. Use a real client identifier (macos-bridge@<ver>) for now.
  • proton.Keys.Primary() panics if no primary key exists. Guard with len(user.Keys) > 0. (We don't currently guard the "keys but no primary" case — every Proton account has one, but if you hit a weird account add the iteration manually.)
  • proton.Auth.TwoFA.Enabled is 0 when 2FA is disabled, otherwise one of the TwoFAStatus constants. HasFIDO2AndTOTP falls back to TOTP transparently; HasFIDO2 alone errors with workaround instructions.
  • modernc.org/sqlite registers its driver as "sqlite" (no 3) and takes pragmas via DSN as ?_pragma=name(value)&_pragma=.... Goose still expects dialect "sqlite3" though — driver name and goose dialect are different concepts and need to be set independently.
  • Goose splits SQL files on ; by default, which breaks CREATE TRIGGER ... BEGIN ... END; blocks (they contain inner semicolons). Wrap each trigger in -- +goose StatementBegin / -- +goose StatementEnd.
